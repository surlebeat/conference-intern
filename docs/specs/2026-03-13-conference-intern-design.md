# Conference Intern — Design Spec

**Date:** 2026-03-13
**Status:** Draft
**Skill:** conference-intern
**Distribution:** ClawhHub (generic, no personal data, no MC dependency)

## Overview

An OpenClaw skill that helps crypto conference attendees discover, curate, and register for side events. It fetches events from Luma and community-curated Google Sheets, filters them based on user preferences using LLM intelligence, and auto-registers on Luma via browser automation.

## Target Users

People attending crypto conferences who want to maximize networking opportunities without missing key side events.

## Pipeline

Four independent, re-runnable stages plus a monitoring wrapper.

### Stage 1: Setup

**Script:** `setup.sh` → `setup-prompt.md`

Interactive setup where the agent walks the user through configuration:

1. Conference name and identifier
2. Luma URLs (event listing pages, calendar pages)
3. Google Sheet links (community-curated side event lists)
4. Interest topics (e.g., "DeFi", "ZK proofs", "infrastructure")
5. Avoid topics (e.g., "NFT minting parties", "token launches")
6. Blocked organizers
7. Registration strategy: aggressive (maximize RSVPs) or conservative (curated only)
8. Monitoring preference: scheduled (with interval), on-demand, or both
9. User info for RSVP: name, email
10. Optional Luma authentication: email 2FA flow via browser (headless-compatible — user pastes code)

**Output:** `conferences/{conference-id}/config.json`

```json
{
  "name": "EthDenver 2026",
  "id": "ethdenver-2026",
  "luma_urls": ["https://lu.ma/ethdenver-side-events"],
  "sheets": ["https://docs.google.com/spreadsheets/d/..."],
  "user_info": {
    "name": "...",
    "email": "..."
  },
  "preferences": {
    "interests": ["DeFi", "ZK proofs", "infrastructure"],
    "avoid": ["NFT minting parties", "token launches"],
    "blocked_organizers": ["SpamDAO"],
    "strategy": "aggressive"
  },
  "monitoring": {
    "mode": "both",
    "interval": "6h"
  },
  "luma_session": {
    "authenticated": false
  }
}
```

**Luma authentication flow (headless-compatible):**

1. Agent opens Luma login page via browser
2. Enters user's email address
3. Luma sends 2FA code to user's email
4. Agent prompts user to paste the code
5. Agent enters code, completes login
6. Session cookies exported and saved to `conferences/{conference-id}/luma-session.json` (kept local to the conference, gitignored)
7. On subsequent runs, cookies loaded before navigating Luma

### Stage 2: Discover

**Script:** `discover.sh`

Fetches events from all configured sources and normalizes into a common schema.

**Google Sheets path:**
1. Try `gog sheets get` with the sheet URL — parse output into events
2. If `gog` unavailable or fails (e.g., public link without auth): try the sheet's "publish to web" CSV export URL (`/pub?output=csv`) via curl
3. If neither works: fall back to browser — open sheet, read content (note: Sheets UI is complex; this is a last resort)

**Luma path:**
1. Agent browser navigates to each URL in `luma_urls`
2. Takes a page snapshot (structured text with interactive elements)
3. LLM reads the snapshot and extracts event data
4. No hardcoded CSS selectors — LLM interprets the page like a human (evergreen approach)

**Event schema** (`events.json`):

```json
[
  {
    "id": "hash-of-name-date",
    "name": "ZK Privacy Summit",
    "date": "2026-03-05",
    "time": "14:00-18:00",
    "location": "Venue Name",
    "description": "...",
    "host": "PrivacyDAO",
    "source": "luma",
    "rsvp_url": "https://lu.ma/...",
    "rsvp_count": 234,
    "is_new": true
  }
]
```

Deduplication by SHA-256 hash of `name+date+time` (truncated to 12 chars) across sources.

### Stage 3: Curate

**Script:** `curate.sh` → `curate-prompt.md`

LLM-driven filtering and ranking:

1. Reads `events.json` + user preferences from `config.json`
2. Scores each event based on:
   - Topic relevance to user interests
   - Host/speaker quality signals
   - RSVP count as a popularity proxy
   - User blocklist (organizers, topics)
3. Applies strategy:
   - **Aggressive:** recommend most events that aren't blocked/avoided
   - **Conservative:** only top-scored events
4. Assigns tiers: must-attend / recommended / optional
5. Lists blocked/filtered events with reason

**Output:** `curated.md`

```markdown
# EthDenver 2026 — Side Events

Last updated: 2026-03-05 14:00 UTC
Strategy: aggressive | Events: 47 found, 31 recommended

## March 5 (Wednesday)

### Must Attend
- **ZK Privacy Summit** — 14:00-18:00 @ Venue Name
  Host: PrivacyDAO | RSVPs: 234
  ✅ Registered

### Recommended
- **DeFi Builders Happy Hour** — 18:00-21:00 @ Bar Name
  Host: DeFi Alliance | RSVPs: 89
  ⏳ Needs input: [Company name, Role]

### Optional
- **Infra Roundtable** — 10:00-12:00 @ Hotel Conf Room
  Host: InfraDAO
  🔗 [Register manually](https://eventbrite.com/...)

## Blocked / Filtered Out
- ~~NFT Minting Party~~ — matched avoid list: "NFT minting parties"
```

### Stage 4: Register

**Script:** `register.sh` → `register-prompt.md`

Browser-based RSVP automation for Luma events:

1. Load Luma session cookies if authenticated
2. For each event marked for registration:
   a. Open the RSVP URL via browser
   b. Read the registration form
   c. Fill standard fields (name, email — prefilled if authenticated)
   d. **Custom field handling:**
      - If unknown/custom fields detected (company, role, wallet address, dietary restrictions, etc.)
      - Mark event as "needs-input" in `curated.md`, listing the required field labels
      - Skip to next event
   e. If all fields filled, submit the form
   f. Confirm registration success
3. After processing all events, notify user about "needs-input" events
4. User provides answers → agent re-runs registration for those specific events
5. Update `curated.md` with registration status per event

**For non-Luma events:** No auto-registration. Listed with manual registration link in `curated.md`.

### Monitoring

**Script:** `monitor.sh`

Re-runs Discover + Curate, diffs against existing `events.json`:

- New events flagged with `is_new: true`
- Removed events noted
- Updated `curated.md` highlights new additions
- Can be triggered on-demand or via scheduled monitoring
- **Scheduled monitoring:** If `monitoring.mode` is `scheduled` or `both`, the setup script creates an OpenClaw cron job via `openclaw cron edit --message "Run conference-intern monitor <conference-id>"` with the configured interval. The agent's heartbeat cycle picks this up and executes `monitor.sh`.
- Keeps `events-previous.json` as a snapshot for diffing; `is_new` is computed by comparing current vs previous run

## Agent Interaction

### Commands

The skill is agent-agnostic. Any agent with browser access can run it. Commands are shell scripts invoked via bash:

| Command | Invocation | Description |
|---------|------------|-------------|
| setup | `bash scripts/setup.sh <conference-name>` | Interactive first-time configuration |
| discover | `bash scripts/discover.sh <conference-id>` | Fetch events from all sources |
| curate | `bash scripts/curate.sh <conference-id>` | Filter and rank events |
| register | `bash scripts/register.sh <conference-id>` | Auto-RSVP to selected events |
| monitor | `bash scripts/monitor.sh <conference-id>` | Discover + curate + flag new events |
| run | `bash scripts/discover.sh <id> && bash scripts/curate.sh <id> && bash scripts/register.sh <id>` | Full pipeline |

`CLAUDE.md` documents these commands so the agent knows how to route user requests to the correct script.

### Prompt Templates

Templates use natural language instructions — no hardcoded browser commands or DOM selectors:

- **setup-prompt.md** — questions to ask, validation rules, how to structure config.json
- **curate-prompt.md** — scoring criteria, strategy application, output format
- **register-prompt.md** — how to navigate forms, fill fields, detect custom fields, when to pause for user input

Example from `register-prompt.md`:
```
1. Open the RSVP URL for the event
2. Read the page — identify the registration form
3. Fill in standard fields: name, email (from config)
4. If you see fields you don't recognize or can't fill, mark the event
   as "needs-input" and list the field labels
5. If all fields are filled, submit the form
6. Confirm registration was successful (look for confirmation message)
```

## CLAUDE.md Structure

`CLAUDE.md` is the primary file the agent reads when using this skill. It contains:

1. **Skill overview** — one-paragraph description of what conference-intern does
2. **Commands** — each command listed with usage, what script it runs, and expected behavior:
   - Commands are invoked as shell scripts: `bash scripts/setup.sh <conference-name>`
   - Each script sources the relevant prompt template and passes data to the LLM
3. **File locations** — where config, events, and curated output live per conference
4. **Browser instructions** — remind the agent to use natural language page interpretation, never hardcode selectors
5. **Error handling rules** — what to do on failure (see Error Handling section)
6. **Stop conditions** — when to pause and ask the user (custom fields, auth codes, zero events)

## Error Handling

Each stage has defined failure behaviors:

**Discover:**
- Luma page fails to load or is rate-limited → log error, skip that URL, continue with remaining sources. Notify user which sources failed.
- Google Sheets: `gog` fails → try CSV export URL → try browser → if all fail, skip sheet and notify user.
- Zero events found across all sources → stop pipeline, notify user ("No events found — check your URLs").

**Curate:**
- Zero events in `events.json` → skip curation, notify user.
- All events filtered out by preferences → notify user, suggest loosening criteria.

**Register:**
- RSVP page fails to load → mark event as "failed" in `curated.md`, continue with next event.
- CAPTCHA detected → mark event as "manual" with link, notify user.
- Event full / registration closed → mark as "closed" in `curated.md`, continue.
- Custom fields detected → mark as "needs-input" (existing flow).
- Session expired mid-run → attempt re-auth if user is interactive, otherwise mark remaining events as "session-expired" and stop.

**Monitor:**
- Same as Discover errors. Additionally, if `events-previous.json` is missing (first run), treat all events as new.

**`run` command short-circuiting:**
- If discover returns zero events → skip curate and register, notify user.
- If curate recommends zero events → skip register, notify user.

## Credential Storage

Follows OpenClaw conventions:

- **Luma session cookies** → `conferences/{conference-id}/luma-session.json` (per-conference, gitignored)
- **User RSVP info** (name, email) → `config.json` per conference (not sensitive)
- **gog CLI** → handles its own auth via system keyring
- **No API keys needed** — all data access is via browser or gog

## Browser Abstraction

The skill does not hardcode any specific browser tool. Prompt templates describe actions in natural language that any LLM with any browser capability can interpret. This makes the skill compatible with:

- agent-browser (Rust CLI)
- Playwright MCP
- Any future browser tool

The SKILL.md metadata declares `browser` as a required capability, letting OpenClaw validate at install time.

## Design Principles

- **Evergreen** — no CSS selectors or DOM assumptions; LLM reads pages like a human
- **Generic** — no personal data, no Mission Control dependency, distributable via ClawhHub
- **Agent-agnostic** — any agent with browser access can execute the skill
- **Re-runnable** — each pipeline stage is idempotent and can be re-run independently
- **User in control** — custom fields always deferred to user, never guessed
